## 시작점

이번에 만들고 싶었던 건 단순 요약기가 아니었습니다. 논문 PDF를 읽고, 핵심 주장과 실험을 정리하고, 한국어 블로그 포스트 형태로 바로 옮길 수 있는 흐름이 필요했습니다. 실제로 계속 쓸 수 있는 도구가 목표였고, 그래서 처음부터 로컬 실행과 블로그 게시를 함께 생각했습니다.

## 왜 따로 만들었는가

논문 리뷰를 손으로 하나씩 정리하면 글맛은 좋지만 시간이 오래 걸립니다. 반대로 자동화만 밀어붙이면 근거가 약하거나, 블로그에 올리기엔 문장이 너무 거칠어지기 쉽습니다. 그래서 이번 툴킷은 그 중간을 노렸습니다. 초안은 빠르게 만들되, 공개 포스트는 사람이 읽기에 자연스럽고 근거도 따라갈 수 있게 만드는 쪽입니다.

## 전체 흐름

툴킷의 기본 흐름은 꽤 단순합니다.

1. PDF 또는 arXiv URL을 입력으로 받습니다.
2. 텍스트를 추출하고, 섹션 구조를 다시 잡습니다.
3. 핵심 주장과 실험 결과를 정리합니다.
4. 한국어 블로그 포스트 형식으로 렌더링합니다.
5. 필요하면 GitHub Pages 블로그 저장소에 반영합니다.

처음에는 그림까지 자동으로 넣는 방향도 같이 시도했는데, 블로그에서 보기에 오히려 어색한 경우가 많았습니다. 그래서 지금은 이미지보다 텍스트와 표를 더 정확하게 만드는 쪽으로 정리했습니다.

## 모델 선택

로컬에서 계속 돌릴 수 있어야 해서, 모델은 Qwen 계열을 기준으로 잡았습니다. M1 Max 32GB 환경에서는 Qwen 14B 정도가 품질과 현실성을 같이 보는 쪽에 더 맞았습니다. 아주 가볍게 돌릴 때는 8B로 내려갈 수 있게 했고, 공개용 자동 게시는 heuristic fallback만으로는 막아두는 안전장치도 넣었습니다.

이 부분은 결국 품질 문제였습니다. 논문 리뷰는 숫자와 주장 하나만 어긋나도 글 전체 신뢰도가 크게 떨어지기 때문에, 대충 빨리보다 어느 정도 안정적인 로컬 추론이 더 중요했습니다.

## 블로그와 툴을 나눈 이유

중간에 한 번 느낀 건, 블로그와 툴이 한 저장소에 있으면 둘 다 애매해진다는 점이었습니다. 블로그는 글이 주인공이어야 하고, 툴 저장소는 설치법과 구조 설명이 주인공이어야 합니다. 그래서 결국 둘을 분리했습니다.

- 블로그 저장소는 공개 글과 페이지 중심
- 툴 저장소는 생성기 코드, 테스트, 입력 데이터 중심

이렇게 나누고 나니 블로그도 훨씬 덜 도구 소개 사이트 같아졌고, 툴도 그 자체로 설명 가능한 형태가 됐습니다.

## 이번에 바꾼 포스트 형식

지금 논문 리뷰 포스트는 조금 더 단순한 형태로 갑니다.

- 그림은 넣지 않음
- 대표 실험은 Markdown 표로 다시 작성
- 본문은 한국어
- 구조는 오버뷰, 핵심 주장, 초록, 서론, 본론, 제안방법, 실험, 결론, 논의

이 방식이 블로그에서 훨씬 읽기 편했습니다. 특히 위장 객체 탐지처럼 표가 중요한 논문은 그림보다 숫자 비교가 더 바로 들어오는 경우가 많았습니다.

## 검수는 어떻게 두었나

공개 블로그에는 검수 로그를 안 보이게 했지만, 내부적으로는 두 번 점검하는 흐름을 남겨두었습니다. 필수 섹션이 비었는지, 숫자가 원문과 맞는지, 과장된 표현이 섞였는지 같은 걸 한 번 더 확인하는 수준입니다.

완벽하게 자동화했다고 말하긴 어렵지만, 적어도 바로 공개해도 되는지 아닌지를 한 번 걸러주는 장치는 있어야 한다고 생각했습니다.

## 지금 상태에서 마음에 드는 점

가장 마음에 드는 건 이제 이 흐름이 실제로 블로그에 연결된다는 점입니다. 논문을 고르고, 정리하고, 페이지에 올리는 일이 한 번의 작업 흐름 안에 들어왔습니다. 그리고 공개 블로그 쪽에서는 도구 얘기를 앞세우지 않고, 그냥 읽히는 글 형태로 남는 것도 좋았습니다.

## 아직 남아 있는 것

아직 완전히 끝난 건 아닙니다. 표 자동 추출은 더 다듬을 여지가 있고, 논문 종류가 바뀌면 지금보다 더 유연한 후처리도 필요합니다. 프로젝트 페이지도 몇 편 더 쌓여야 훨씬 자연스러워질 겁니다. 그래도 지금 상태면, 실제로 계속 쓰면서 발전시키기엔 충분히 괜찮은 첫 버전이라고 생각합니다.
